# Referencia de Endpoints

## Grupos de Endpoints

### Health

Endpoints de monitoreo que verifican el estado de la aplicacion y sus dependencias. Utilizados por Kubernetes para liveness/readiness probes y por sistemas de monitoreo externos.

| Metodo | Ruta | Auth | Descripcion |
|--------|------|------|-------------|
| GET | `/api/v1/health` | No | Health check completo con estado de BD |
| GET | `/api/v1/health/ready` | No | Readiness probe para K8s |
| GET | `/api/v1/health/live` | No | Liveness probe para K8s |

### Auth

Gestion de autenticacion y sesiones de usuario. Incluye registro, login, refresh de tokens y cierre de sesion.

| Metodo | Ruta | Auth | Descripcion |
|--------|------|------|-------------|
| POST | `/api/v1/auth/register` | No | Registrar nuevo usuario |
| POST | `/api/v1/auth/login` | No | Iniciar sesion (rate limited: 5/15min) |
| POST | `/api/v1/auth/refresh` | No | Renovar tokens |
| POST | `/api/v1/auth/logout` | Si | Cerrar sesion |
| GET | `/api/v1/auth/profile` | Si | Obtener perfil del usuario autenticado |

### Users

Gestion de usuarios del sistema. Las operaciones de escritura requieren rol ADMIN.

| Metodo | Ruta | Auth | Rol | Descripcion |
|--------|------|------|-----|-------------|
| GET | `/api/v1/users` | Si | ADMIN | Listar usuarios (paginado) |
| GET | `/api/v1/users/:id` | Si | USER/ADMIN | Obtener usuario por ID |
| PATCH | `/api/v1/users/:id` | Si | ADMIN | Actualizar usuario |
| DELETE | `/api/v1/users/:id` | Si | ADMIN | Eliminar usuario |
| PATCH | `/api/v1/users/:id/active` | Si | ADMIN | Activar/desactivar usuario |

### Activity Log

Auditoria completa de todas las actividades del sistema. Solo accesible por administradores.

| Metodo | Ruta | Auth | Rol | Descripcion |
|--------|------|------|-----|-------------|
| GET | `/api/v1/activity-log` | Si | ADMIN | Listar actividades (paginado + filtros) |
| GET | `/api/v1/activity-log/:id` | Si | ADMIN | Actividad por ID |
| GET | `/api/v1/activity-log/user/:userId` | Si | ADMIN | Actividades de un usuario |
| GET | `/api/v1/activity-log/stats` | Si | ADMIN | Estadisticas de actividad |
| GET | `/api/v1/activity-log/critical` | Si | ADMIN | Actividades criticas recientes |
| POST | `/api/v1/activity-log/export` | Si | ADMIN | Exportar a CSV o JSON |

### Ops

Dashboard de operaciones con metricas del sistema, estado de servicios, colas y contenedores.

| Metodo | Ruta | Auth | Rol | Descripcion |
|--------|------|------|-----|-------------|
| GET | `/api/v1/ops` | Si | ADMIN | Dashboard completo de operaciones |

### Metrics

Endpoint de Prometheus para recoleccion de metricas de observabilidad.

| Metodo | Ruta | Auth | Descripcion |
|--------|------|------|-------------|
| GET | `/api/v1/metrics` | API Key | Metricas Prometheus (text/plain) |

## Paginacion

Los endpoints de listado utilizan un formato de paginacion consistente:

### Parametros de consulta

| Parametro | Tipo | Default | Min | Max | Descripcion |
|-----------|------|---------|-----|-----|-------------|
| `page` | integer | 1 | 1 | - | Numero de pagina |
| `limit` | integer | 10 (users) / 20 (activity) | 1 | 100 | Items por pagina |

### Formato de respuesta

```json
{
  "data": [],
  "meta": {
    "total": 150,
    "page": 1,
    "limit": 10,
    "totalPages": 15
  }
}
```

Nota: Algunos endpoints pueden usar un formato plano en lugar de `data`/`meta`:

```json
{
  "users": [],
  "total": 150,
  "page": 1,
  "limit": 10
}
```

## Formato de Errores

Todos los endpoints devuelven errores en un formato estandar:

```json
{
  "statusCode": 400,
  "message": "Descripcion del error",
  "error": "Bad Request",
  "timestamp": "2026-07-04T12:00:00.000Z",
  "path": "/api/v1/auth/register"
}
```

### Codigos de error comunes

| Codigo | Significado | Causa tipica |
|--------|-------------|--------------|
| 400 | Bad Request | Validacion fallida (DTO invalido) |
| 401 | Unauthorized | Token faltante, expirado o invalido |
| 403 | Forbidden | Rol insuficiente (no ADMIN) |
| 404 | Not Found | Recurso no existe |
| 409 | Conflict | Email duplicado en registro |
| 413 | Payload Too Large | Body excede 1MB |
| 429 | Too Many Requests | Rate limit excedido |
| 500 | Internal Server Error | Error inesperado del servidor |

### Validacion (400)

Cuando un DTO falla la validacion, `message` puede ser un array con los errores especificos:

```json
{
  "statusCode": 400,
  "message": [
    "email must be an email",
    "password must be longer than or equal to 8 characters"
  ],
  "error": "Bad Request",
  "timestamp": "2026-07-04T12:00:00.000Z",
  "path": "/api/v1/auth/register"
}
```

## Headers de Seguridad

Todas las respuestas incluyen estos headers de seguridad:

| Header | Valor |
|--------|-------|
| `Content-Security-Policy` | `default-src 'self'; script-src 'self' 'unsafe-inline'; ...` |
| `X-Content-Type-Options` | `nosniff` |
| `Strict-Transport-Security` | `max-age=31536000; includeSubDomains; preload` |
| `X-Frame-Options` | `DENY` |
| `Referrer-Policy` | `strict-origin-when-cross-origin` |
| `X-XSS-Protection` | `1; mode=block` |
| `X-Robots-Tag` | `noindex, nofollow` (en metrics) |

## Rate Limiting por Endpoint

| Endpoint | Tier | Limite | TTL |
|----------|------|--------|-----|
| `POST /api/v1/auth/login` | Especial | 5 | 15 minutos |
| `POST /api/v1/auth/register` | short | 3 | 1 segundo |
| `POST /api/v1/auth/logout` | short | 3 | 1 segundo |
| `GET /api/v1/auth/profile` | medium | 20 | 10 segundos |
| `GET /api/v1/users` | medium | 20 | 10 segundos |
| `GET /api/v1/activity-log/*` | medium | 20 | 10 segundos |
| `GET /api/v1/health/*` | long | 100 | 60 segundos |
| `GET /api/v1/ops` | medium | 20 | 10 segundos |
